## 一、项目概述

### 1.1 项目定位

QuickOJ 是一个快速、简洁的算法竞赛在线评测系统，提供在线刷题、代码提交、自动评测、题目审核、排行榜等综合性功能。

```
│  │                    → 轮询结果→回写 DB        │   │
│  └────────────────────────────────────────────┘   │

       │ pyodbc                     │ HTTP

┌──────────────┐          ┌──────────────────┐
│  SQL Server   │          │  评测机 (独立进程) │
│  OJPlatform   │          │  localhost:8001   │
│               │          │  POST /judge      │
│               │          │  GET  /job/{id}   │

└──────────────┘          └──────────────────┘
```

### 3.2 数据流（代码提交→评测→结果回写）

```
1. 用户 POST /problems/{id}/submit (code, language)
2. 后端创建 Submission(status='pending')，存入 DB
3. 评测调度器 (JudgeDispatcher._poll) 每 2s 轮询:
   a. 检查 JudgeState.is_connected 和 heartbeat
   b. SELECT ... WHERE status='pending' ORDER BY is_test_run DESC, submitted_at ASC
   c. 该条 status → 'running'
   d. 【后端】根据 is_test_run 决定数据源:
      - is_test_run=0 → 读 PROBLEMS_DATA_DIR/{problem_id}/ 下的 .in/.out 文件
      - is_test_run=1 → 读 PROBLEMS_PENDING_DIR/{problem_id}/ 下的 .in/.out 文件
   e. 【后端】将代码+测试数据文件打包，HTTP POST /judge 发送给评测机
      (multipart: code + language + time_limit_ms + memory_limit_kb + test_data 文件列表)
   f. 评测机返回 job_id，后端注册到 _pending_jobs
4. 评测调度器 (JudgeDispatcher._check_pending_jobs) 每 2s:
   a. GET /job/{job_id} 轮询结果
   b. status='done' → _write_result() 回写 Submission 表
   c. 更新 User.solved_problems, Problem.accepted_user_count 等统计
5. 前端 SubmissionDetail 页每 3s 轮询 GET /submissions/{id} 直到状态为终态

关键：评测机不访问数据库，也不直接读取磁盘上的测试数据文件。
     所有数据（代码 + 测试用例）均由后端从磁盘读取后，通过 HTTP multipart 发送给评测机。
```


### 3.3 调试数据流

```
1. 用户 POST /problems/{id}/debug (code, language, input_data)
2. 后端直接 HTTP POST /judge 发给评测机（不经过调度器，不存 DB）:
   (multipart: code + language + is_debug=true + debug_input + time_limit_ms + memory_limit_kb)
   注意：调试模式不发送 test_data 文件，仅发送用户的自定义输入 debug_input
3. 评测机在临时目录创建 input.txt（内容 = debug_input）+ answer.txt（空），运行用户代码
4. 评测机返回 job_id，后端注册 _pending_jobs[job_id] = -1（负数标记为调试任务）
5. 前端每 1s 轮询 GET /problems/debug/result/{job_id}
6. 结果存入 _debug_results（内存字典），前端获取后不写入 DB，不创建 Submission 记录
```

## 四、用户角色与权限

### 4.1 角色定义

```python
ROLE_LEVEL = {"root": 1, "admin": 2, "user": 3}
```

| 角色 | 标识 | 权限等级 | 说明 |
| :--- | :--- | :--- | :--- |
| 站长 | `root` | 1（最高） | 第一个注册用户自动成为站长。系统中至少保留一个 root。 |
| 管理员 | `admin` | 2 | 由站长任命或撤销 |
| 普通用户 | `user` | 3（最低） | 默认注册角色 |

### 4.2 权限矩阵

| 功能 | user | admin | root |
| :--- | :--- | :--- | :--- |
| 浏览公开题目 | ✅ | ✅ | ✅ |
| 提交代码 | ✅ | ✅ | ✅ |
| 代码调试 | ✅ | ✅ | ✅ |
| 上传题目 | ✅→pending_new | ✅→pending_new | ✅→pending_new |
| 修改自己的题目 | ✅ | ✅ | ✅ |
| 删除自己的未公开题 | ✅ | ✅ | ✅ |
| 删除已公开题 | ❌ | ✅ | ✅ |
| 审核题目 | ❌ | ✅ | ✅ |
| 冻结/归档题目 | ❌ | ✅ | ✅ |
| 发送系统消息 | ❌ | ✅ | ✅ |
| 管理评测机连接 | ❌ | ✅ | ✅ |
| 封禁/解封用户 | ❌ | ✅（权限低于自己） | ✅ |
| 修改用户角色 | ❌ | ❌ | ✅ |
| 查看全量提交记录 | ❌ | ✅ | ✅ |

### 4.3 权限保护规则

1. **封禁保护**：操作者 `ROLE_LEVEL` 必须低于（数字更小）目标用户的 `ROLE_LEVEL`，否则拒绝
2. **角色修改保护**：仅 `root` 可操作；不可修改自己；不可将最后一个 `root` 降级
3. **自我操作保护**：不可封禁自己、不可修改自己的角色
4. **状态保护**：封禁用户时自动 `token_version += 1`，使其所有 Token 立即失效

### 4.4 认证机制

- **密码哈希**：bcrypt（passlib）
- **Access Token**：JWT，有效期 24h，Payload = `{user_id, role, token_version, type:"access", exp}`
- **Refresh Token**：JWT，有效期 7 天，Payload = `{user_id, token_version, type:"refresh", exp}`
- **单设备登录**：每次登录 `token_version += 1`，旧 Token 版本号不匹配→401
- **Token 刷新**：Refresh Token 换取新 Access Token（校验版本号）
- **封禁处理**：`get_current_user` 依赖检查 `user.status == "banned"` → 401

## 五、数据库设计

### 5.1 完整建表 SQL

```sql
-- ============================================================
-- 1. Users 表 —— 用户身份、权限、统计
-- ============================================================
CREATE TABLE Users (
    user_id          BIGINT IDENTITY(1,1) PRIMARY KEY,   -- 用户唯一ID，自增
    username         VARCHAR(30)    NOT NULL,             -- 登录用户名，4~20位字母/数字/下划线，首字不能是数字
    password_hash    VARCHAR(255)   NOT NULL,             -- bcrypt 密码哈希值，不存明文
    role             VARCHAR(20)    NOT NULL DEFAULT 'user'  -- 角色：root(站长) / admin(管理员) / user(普通用户)
                        CONSTRAINT CK_Users_role CHECK (role IN ('root','admin','user')),
    status           VARCHAR(20)    NOT NULL DEFAULT 'active'  -- 账号状态：active(正常) / banned(已封禁)
                        CONSTRAINT CK_Users_status CHECK (status IN ('active','banned')),
    email            VARCHAR(100)   NULL,                 -- 邮箱地址，可选，需唯一
    avatar           VARCHAR(255)   NULL,                 -- 头像相对路径，如 /avatars/1_a1b2c3d4.jpg；UUID命名以绕过浏览器缓存
    solved_problems  INT            NOT NULL DEFAULT 0,   -- 已通过题目总数（冗余字段，首次AC时+1）
    total_submissions INT           NOT NULL DEFAULT 0,   -- 总正式提交次数（冗余字段，每次提交+1）
    token_version    INT            NOT NULL DEFAULT 0,   -- Token版本号：每次登录/改密/被封时+1，旧Token立即失效
    registered_at    DATETIME       NOT NULL DEFAULT GETDATE(),  -- 注册时间（UTC）
    last_login       DATETIME       NULL,                 -- 最后登录时间（UTC）
    CONSTRAINT UQ_Users_username UNIQUE (username)
);

-- ============================================================
-- 2. Tags 表 —— 题目标签主表
-- ============================================================
CREATE TABLE Tags (
    tag_id      BIGINT IDENTITY(1,1) PRIMARY KEY,   -- 标签唯一ID，自增
    tag_name    NVARCHAR(50)  NOT NULL,              -- 标签名称（中文/英文），≤8字符
    description NVARCHAR(500) NULL,                  -- 标签描述，可选
    problems    BIGINT        NOT NULL DEFAULT 0,    -- 关联题目数量（冗余字段，增删标签时同步更新）
    CONSTRAINT UQ_Tags_tag_name UNIQUE (tag_name)
);

-- ============================================================
-- 3. Problems 表 —— 题目信息、状态、审核记录
-- ============================================================
CREATE TABLE Problems (
    problem_id                   BIGINT IDENTITY(1,1) PRIMARY KEY,  -- 题目唯一ID，自增；同时作为数据文件夹名
    problem_number               BIGINT        NULL,                -- 对外展示题号，仅审核通过后分配（起始1001）；未通过时为NULL
    uploader_id                  BIGINT        NOT NULL REFERENCES Users(user_id),  -- 上传者ID
    problem_status               VARCHAR(20)   NOT NULL DEFAULT 'pending_new'  -- 题目状态（详见附录C）
                                    CONSTRAINT CK_Problems_status CHECK (problem_status IN
                                        ('pending_new','pending_modify','approved','rejected','archived','frozen')),
    problem_name                 NVARCHAR(50)  NOT NULL,            -- 题目标题，≤50字符
    difficulty                   INT           NOT NULL,            -- 难度分数（如800、1200、2000），无单位
    statement                    NVARCHAR(MAX) NOT NULL,            -- 题面描述，Markdown格式
    problem_type                 VARCHAR(20)   NOT NULL             -- 题型，创建后不可修改
                                    CONSTRAINT CK_Problems_type CHECK (problem_type IN
                                        ('traditional','interactive','output-only','communication')),
    time_limit                   INT           NOT NULL,            -- 时间限制，单位 ms
    memory_limit                 INT           NOT NULL,            -- 内存限制，单位 KB
    accepted_user_count          INT           NOT NULL DEFAULT 0,  -- 通过此题的不同用户数（冗余字段，首次AC时+1）
    submissions_before_accepted  INT           NOT NULL DEFAULT 0,  -- 首次通过前的总提交次数（含非AC提交）
    source                       NVARCHAR(100) NULL,                -- 题目来源，如 LeetCode、洛谷
    sample_download_policy       VARCHAR(20)   NOT NULL DEFAULT 'none'  -- 样例下载策略
                                    CONSTRAINT CK_Problems_download CHECK (sample_download_policy IN
                                        ('all','none','first_failed')),
    created_at                   DATETIME      NOT NULL DEFAULT GETDATE(),  -- 创建时间（UTC）
    updated_at                   DATETIME      NULL,                -- 最后修改时间（每次进入pending状态时更新）
    reviewer_id                  BIGINT        NULL REFERENCES Users(user_id),  -- 审核人ID，未审核时为NULL
    reviewed_at                  DATETIME      NULL,                -- 审核时间（UTC）
    review_comment               NVARCHAR(500) NULL,                -- 审核备注/拒绝理由（≤500字符）
    CONSTRAINT UQ_Problems_number UNIQUE (problem_number)           -- 题号全局唯一
);

-- ============================================================
-- 4. TagProblems 表 —— 题目与标签的多对多关联
-- ============================================================
CREATE TABLE TagProblems (
    tag_id     BIGINT NOT NULL REFERENCES Tags(tag_id),         -- 标签ID（复合主键）
    problem_id BIGINT NOT NULL REFERENCES Problems(problem_id), -- 题目ID（复合主键）
    PRIMARY KEY (tag_id, problem_id)
);

-- ============================================================
-- 5. Submissions 表 —— 提交记录与评测结果
-- ============================================================
CREATE TABLE Submissions (
    submission_id  BIGINT IDENTITY(1,1) PRIMARY KEY,  -- 提交唯一ID，自增
    user_id        BIGINT         NOT NULL,            -- 提交者ID（不设外键以保留用户删除后的记录）
    username       NVARCHAR(50)   NULL,                -- 提交者用户名（冗余字段，避免JOIN Users）
    problem_id     BIGINT         NOT NULL,            -- 题目ID
```

```sql
-- 待审核题目队列

SELECT problem_id, problem_number, problem_name, problem_status, difficulty,
       problem_type, uploader_id, updated_at, created_at
FROM Problems
WHERE problem_status IN ('pending_new', 'pending_modify');

SELECT uap.user_id, uap.problem_id, uap.first_accepted_at,

FROM UserAcceptedProblems uap
JOIN Problems p ON p.problem_id = uap.problem_id;
```

### 5.4 设计要点说明

**反范式冗余**：Submissions 表冗余了 `username`、`problem_number`、`problem_name`、`problem_type`、`code_length` 五个字段。目的是避免提交记录列表页（高频查询）每次 JOIN Users 和 Problems 表，这是有意的性能优化。

**统计冗余**：`Users.solved_problems`、`Users.total_submissions`、`Problems.accepted_user_count`、`Problems.submissions_before_accepted`、`Tags.problems` 均为冗余计数字段，由后端在相关操作时同步更新，以提升查询性能。

## 六、API 接口设计

### 6.1 通用规范

**Base URL**：开发环境 `http://localhost:8000`，生产环境通过 Cloudflare Tunnel 暴露

**Content-Type**：
- JSON 接口：`application/json`
- 文件上传：`multipart/form-data`（后端使用 Form 字段而非 JSON Body）

**认证方式**：请求头 `Authorization: Bearer <access_token>`

**统一响应格式**：
```json
// 成功

// 失败
{ "code": 1, "message": "错误描述", "data": null }

// 分页
{
  "code": 0, "message": "操作成功",
  "data": {
    "list": [ ... ],
    "pagination": { "page": 1, "size": 20, "total": 100, "total_pages": 5 }
  }
}
```

**分页参数**：`page`（默认 1）、`size`（默认 20，最大 100）

**HTTP 错误码**：400（请求参数错误）、401（未认证/token_version不匹配/被封禁）、403（权限不足）、404（资源不存在）、409（冲突）

### 6.2 用户接口（11 个端点，路由前缀 `/user`）

#### POST `/user/register` — 注册

- **认证**：否
- **Content-Type**：`application/json`
- **请求体**：
  ```json
  { "username": "string (4-20位, 字母/数字/下划线, 首字不能是数字)",
    "password": "string (6-32位)",
    "email": "string (可选, 有效邮箱格式)" }
  ```
- **后端校验**：Pydantic `RegisterRequest`：username 正则 `^[a-zA-Z_][a-zA-Z0-9_]*$`，password 6-32 位，email 可选 EmailStr
- **业务逻辑**：检查用户名/邮箱唯一性 → 第一个用户自动 `role='root'` → bcrypt 哈希密码 → 插入 Users 表
- **成功响应**：`{ "code": 0, "message": "注册成功", "data": { "user_id": 1001, "username": "..." } }`
- **失败响应**：`{ "code": 1, "message": "用户名已存在" }` 或 `"邮箱已被注册"`

#### POST `/user/login` — 登录

- **认证**：否
- **Content-Type**：`application/json`
- **请求体**：`{ "username": "string", "password": "string" }`
- **业务逻辑**：查用户 → 验证密码（bcrypt）→ 检查是否被封禁 → `token_version += 1` → `last_login = now` → 生成 Access Token + Refresh Token
- **成功响应**：
  ```json
  { "code": 0, "message": "登录成功",
    "data": {
      "token": "eyJ...", "refresh_token": "eyJ...", "expires_in": 86400,
      "user": { "user_id": 1001, "username": "...", "avatar": "...", "role": "user",
                "solved_problems": 42, "total_submissions": 128 }
    }
  }
  ```
- **失败响应**：`"用户名或密码错误"` / `"账号已被封禁，请联系管理员"`

#### GET `/user/me` — 获取当前用户信息

- **认证**：Bearer Token
- **响应 data**：`{ user_id, username, email, avatar, role, status, solved_problems, total_submissions, registered_at, last_login }`

#### POST `/user/refresh` — 刷新令牌

- **认证**：否
- **Content-Type**：`application/json`
- **请求体**：`{ "refresh_token": "string" }`
- **业务逻辑**：解码 Refresh Token → 验证 type="refresh" → 验证 token_version 匹配 → 生成新 Access Token
- **成功响应**：`{ "code": 0, "message": "刷新成功", "data": { "token": "...", "expires_in": 86400 } }`
- **失败响应**：`"令牌已过期，请重新登录"` / `"令牌类型错误"` / `"令牌已失效，请重新登录"`

#### POST `/user/logout` — 登出

- **认证**：Bearer Token
- **业务逻辑**：无状态方案，后端不处理（直接返回成功），前端清除 localStorage
- **响应**：`{ "code": 0, "message": "登出成功", "data": null }`

#### PUT `/user/{user_id}` — 更新用户信息

- **认证**：Bearer Token（仅本人）
- **Content-Type**：`application/json`
- **请求体**：`{ "username?": "string", "email?": "string" }`（至少提供一个）
- **业务逻辑**：验证唯一性 → 更新字段 → 不影响 token_version
- **成功响应**：`{ "code": 0, "message": "更新成功", "data": { "user_id": ..., "username": "...", "email": "...", "avatar": "..." } }`

#### PUT `/user/{user_id}/avatar` — 上传头像

- **认证**：Bearer Token（仅本人）
- **Content-Type**：`multipart/form-data`
- **请求体**：`avatar` (file, jpg/png/gif/webp, ≤5MB)
- **业务逻辑**：验证文件类型和大小 → UUID 文件名 `{user_id}_{uuid8}.{ext}` → 删除旧头像文件（glob 匹配 `{user_id}_*`）→ 写入磁盘 → 更新 DB avatar 字段
- **成功响应**：`{ "code": 0, "message": "头像更新成功", "data": { "user_id": ..., "avatar": "/avatars/1_a1b2c3d4.jpg" } }`

#### PUT `/user/{user_id}/password` — 修改密码

- **认证**：Bearer Token（仅本人）
- **Content-Type**：`application/json`
- **请求体**：`{ "old_password": "string", "new_password": "string (6-32位)" }`
- **业务逻辑**：验证旧密码 → bcrypt 哈希新密码 → `token_version += 1` → 返回新 Access Token（当前设备保持登录，其他设备 Token 失效）
- **成功响应**：`{ "code": 0, "message": "密码修改成功", "data": { "token": "...", "expires_in": 86400 } }`

#### GET `/user/{user_id}` — 获取用户公开信息

- **认证**：否
- **响应 data**：`{ user_id, username, avatar, role, solved_problems, total_submissions, registered_at, recent_accepted: [...] }`
- **recent_accepted**：最近通过的 5 道题（通过视图 `v_user_solved` 查询），每项含 `problem_id, problem_number, problem_name, first_accepted_at`

#### GET `/user/{user_id}/solved` — 获取用户通过的题目列表

- **认证**：否
- **分页**：是
- **响应 list 项**：`{ problem_id, problem_number, problem_name, difficulty, first_accepted_at }`
- **数据源**：视图 `v_user_solved`

#### GET `/user/{user_id}/problems` — 获取用户发布的题目列表

- **认证**：可选（未登录仅见公开/重新审核题；本人或管理员见全部）
- **分页**：是
- **响应 list 项**：`{ problem_id, problem_number, problem_name, problem_status, created_at }`

### 6.3 管理接口（8 个端点，路由前缀 `/admin`）

#### GET `/admin/users` — 获取用户列表

- **认证**：Bearer Token（admin+）
- **查询参数**：`role?` (root/admin/user), `status?` (active/banned), `keyword?` (模糊匹配 username), `page?`, `size?`
- **响应 list 项**：`{ user_id, username, role, status, solved_problems, total_submissions, registered_at }`

#### PUT `/admin/users/{user_id}/role` — 修改用户角色（站长专用）

- **认证**：Bearer Token（root only）
- **查询参数**：`role` (admin/user；不可设为 root)
- **业务逻辑**：不可改自己 → 不可将最后一个 root 降级 → 更新角色 → 发通知给目标用户
- **成功响应**：`{ "code": 0, "message": "角色修改成功", "data": { "user_id": ..., "username": "...", "role": "admin" } }`

#### PUT `/admin/users/{user_id}/status` — 封禁/解封用户

- **认证**：Bearer Token（admin+）
- **查询参数**：`status` (active/banned), `reason?` (封禁原因)
- **业务逻辑**：不可改自己 → 不可改权限不低于自己的用户 → 封禁时 `token_version += 1` → 发通知
- **成功响应**：`{ "code": 0, "message": "用户已封禁", "data": { "user_id": ..., "username": "...", "status": "banned" } }`

#### PUT `/admin/problems/{problem_id}/status` — 更改题目状态（审核）

- **认证**：Bearer Token（admin+）
- **查询参数**：`status` (approved/rejected/frozen/archived；不可 pending_new/pending_modify), `review_comment?`
- **业务逻辑**：
  - **approved**：若 `problem_number IS NULL` 则分配新编号（max+1，起始 1001）；已有编号则保留 → `reviewer_id=当前用户` → `reviewed_at=now` → 延迟移动测试数据文件 → 通知上传者
  - **rejected**：若原状态为 `pending_modify` → 回退为 `approved`（题目恢复公开）；否则设为 `rejected` → 延迟清理待审核文件 → 通知上传者（含拒绝理由）
  - **frozen/archived**：直接改状态
- **延迟文件操作**：审核通过时注册 `defer_file_op(problem_id, "approve")`；拒绝时注册 `defer_file_op(problem_id, "reject", is_modify)`。评测调度器每轮检查：该题 `running` 计数为 0 时执行。

#### POST `/admin/notifications` — 发送系统消息

- **认证**：Bearer Token（admin+）
- **查询参数**：`title` (≤100字符), `content` (≤500字符), `target_user_id?` (按 ID 指定) 或 `target_username?` (按用户名指定)（二选一）
- **成功响应**：`{ "code": 0, "message": "消息已发送给用户「xxx」", "data": { "recipient_id": ..., "username": "...", "created_at": "..." } }`

#### 评测机管理（3 个端点，路由前缀 `/admin/judge`）

**GET `/admin/judge`** — 获取评测机状态
- **响应**：`{ "running": bool (heartbeat < 10s), "connected": bool, "status": "idle"|"stopped" }`

**POST `/admin/judge/connect`** — 连接评测机
- **查询参数**：`cutoff?` (ISO datetime；早于此时间的 pending 提交标为 rj)
- **业务逻辑**：检查 heartbeat 存在且 ≤30s → `is_connected = True` → 可选拒绝旧提交
- **响应**：`{ "code": 0, "message": "已连接，N 条旧提交被拒绝" }`

**POST `/admin/judge/disconnect`** — 断开评测机
- **业务逻辑**：`is_connected = False` → 所有 `running` 提交 → `system_error`
- **响应**：`{ "code": 0, "message": "已断开，N 条被中断" }`

- **路径参数**：`p{id}` → 按 problem_id 查；纯数字 → 按 problem_number 查
- **权限**：`can_access_problem` 函数检查（approved/pending_modify→所有人；pending_new/rejected→上传者/管理员；frozen→管理员可见题面；archived→仅管理员）
- **响应 data**：`{ problem_id, problem_number, problem_name, problem_type, problem_status, has_pending_data, test_cases: [...], is_new_problem, difficulty, time_limit, memory_limit, statement, source, sample_download_policy, tags: [...], uploader: {user_id, username}, statistics: {submissions_before_accepted, accepted_user_count}, created_at, updated_at, user_status: {is_solved, last_submission_status} }`

  1. 验证题目存在且可提交（approved/pending_modify，或管理员+pending_new）
  2. 管理员使用待审核数据需 `has_pending_data(problem_id)==True`

  4. 返回 submission_id，评测由后台调度器异步处理
- **成功响应**：`{ "code": 0, "message": "提交成功，正在评测", "data": { "submission_id": 12345, "status": "pending" } }`

- **认证**：Bearer Token
- **Content-Type**：`multipart/form-data`
- **Form 字段**：`code` (string), `language` (cpp/python3), `input_data` (string, 自定义输入)
- **业务逻辑**：检查评测机在线 → HTTP POST `/judge` (is_debug=true) → 返回 job_id → `_pending_jobs[job_id] = -1`
- **成功响应**：`{ "code": 0, "message": "调试请求已提交", "data": { "job_id": "a1b2c3d4e5f6" } }`

#### GET `/problems/debug/result/{job_id}` — 获取调试结果

- **认证**：Bearer Token
- **业务逻辑**：查 `_debug_results[job_id]`，有则返回并删除；无但在 `_pending_jobs` 中→返回 `status:'pending'`；都不在→`system_error`
- **响应 data**：`{ output, error, run_time, run_memory, status (ok/compile_error/time_limit_exceeded/memory_limit_exceeded/runtime_error/system_error/pending), compile_error? }`

#### POST `/problems` — 创建题目

- **认证**：Bearer Token
- **Content-Type**：`multipart/form-data`
- **Form 字段**：`problem_name`* (≤50字), `statement`* (Markdown), `problem_type`* (创建后不可改), `difficulty`* (int), `time_limit`* (ms), `memory_limit`* (KB), `sample_download_policy`* (all/none/first_failed), `source?`, `tags?` (JSON 数组字符串, 如 `"[1,3,5]"`), `test_data` (多文件 .in/.out), `images` (多文件)
- **业务逻辑**：创建 Problem(status='pending_new') → flush 获取 problem_id → 关联标签 → 保存测试数据到 `pending/` 目录 → 保存题面图片
- **成功响应**：`{ "code": 0, "message": "上传成功，等待管理员审核", "data": { "problem_id": ..., "problem_number": null, "problem_status": "pending_new" } }`

#### PUT `/problems/{problem_id}` — 修改题目

- **认证**：Bearer Token（上传者或管理员）
- **Content-Type**：`multipart/form-data`
- **Form 字段**：`problem_name?`, `statement?`, `difficulty?`, `time_limit?`, `memory_limit?`, `sample_download_policy?`, `source?`, `tags?`, `test_data?` (多文件), `excluded_roots?` (JSON 数组, 标记排除的旧样例), `images?` (多文件)
- **业务逻辑**：
  - 如有 test_data 或 excluded_roots：先 `copy_approved_to_pending(problem_id, excluded)` 复制旧数据到 pending 目录
  - 更新字段（problem_type 被忽略）
  - 如有 test_data：已通过题→状态变 `pending_modify`；被拒绝题→状态变 `pending_new`
  - 保存新测试数据（append 模式合并到 pending）
  - 管理员编辑他人题目→通知上传者
- **成功响应**：`{ "code": 0, "message": "修改成功" }` 或 `"...测试数据已更新，需等待管理员重新审核" }`

#### DELETE `/problems/{problem_id}` — 删除题目

- **认证**：Bearer Token（上传者可删未公开题；管理员可删所有）
- **业务逻辑**：删除 TagProblems 关联 → 删除 UserAcceptedProblems 关联 → 删除 Problem 记录 → `delete_problem_data(problem_id)` 删磁盘文件（失败不影响）

#### DELETE `/problems/{problem_id}/testdata/{root}` — 删除测试样例

- **业务逻辑**：删除 pending 目录中 `{root}.in` 和 `{root}.out`/`{root}.ans`

## 七、评测机架构

### 7.1 设计原则

评测机是完全独立于后端的 HTTP 服务：
- **零后端依赖**：不 `import` 任何后端代码
- **零数据库访问**：代码和测试数据均通过 HTTP multipart 传入
- **零持久化**：每次评测在 `tempfile.mkdtemp()` 中完成，事后自动删除
- **进程隔离**：`ProcessPoolExecutor(max_workers = cpu_count - 1)`，每个任务独立子进程
- **跨平台**：Windows（Win32 API）和 Linux（/proc）双内存监控路径

### 7.2 评测机 API

| 端点 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/health` | GET | 健康检查 → `{"status":"ok","workers":N}` |
| `/status` | GET | 运行统计 → `{"running":N,"queued":N,"total_jobs":N,"workers":N}` |
| `/judge` | POST | 提交任务（multipart）→ `{"job_id":"...","status":"queued"}` |
| `/job/{job_id}` | GET | 查询结果 → `{"status":"queued|running|done","result":{...}}` |

**POST `/judge` 请求参数（multipart form）**：

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `code` | string | 源代码 |
| `language` | string | cpp / python3 |
| `problem_id` | int | 题目 ID |
| `time_limit_ms` | int | 时间限制（ms） |
| `memory_limit_kb` | int | 内存限制（KB） |
| `is_test_run` | bool | 是否为审核测试提交 |
| `is_debug` | bool | 是否为调试模式 |
| `debug_input` | string | 调试自定义输入 |
| `test_data` | file[] | `.in`/`.out`/`.ans` 文件列表 |

**评测结果结构（`result`）**：

```json
{
  "summary": { "status": "Accepted", "total": 3, "passed": 3, "failed": 0 },
  "tests": [
    { "name": "1", "status": "Accepted", "time_used": 0.012, "memory_used_kb": 2560, "program_output": "...", "message": "答案正确" }
  ],
  "run_time": 12,
  "run_memory": 2560,
  "compile": { "status": "OK", "output": "" }
}
```

### 7.3 评测流程（工作进程内）

1. 创建临时目录 `tempfile.mkdtemp(prefix="js_worker_")`
2. 写入源代码文件（`main.cpp` 或 `main.py`）
3. 写入从 HTTP 请求收到的测试数据文件到 `data/` 子目录
4. **C++ 编译**：`g++ -O2 -std=c++17 -static -x c++ -`（从 stdin 读取源码，默认输出 `a.exe` 后重命名为 `main.exe`），30s 超时
5. **调试模式**：创建 `input.txt`（用户输入）+ `answer.txt`（空），运行一次
6. **正式评测**：`_discover_test_cases` 配对 .in/.out → 逐个运行 `_run_single_test`
7. **单次运行**：`subprocess.Popen` + 后台内存监控线程（每 0.1s 采样） + 独立 stdout/stderr 读取线程 + `proc.wait(timeout=time_limit_s)`
8. **比对**：normalize（去行尾空格+去末尾空行）→ 完全一致=AC → token 一致=PE → 否则=WA
9. 汇总结果 → 清理临时目录

### 7.4 内存监控实现

**Windows**：`psapi.GetProcessMemoryInfo` + `PROCESS_MEMORY_COUNTERS_EX.PrivateUsage`（当前）/ `PeakWorkingSetSize`（峰值）

**Linux**：`/proc/{pid}/statm` 读取 RSS

### 7.5 测试样例发现

按优先级匹配文件名模式：
`{n}.in/.out` → `input{n}.txt/output{n}.txt` → `in{n}.txt/out{n}.txt` → `test{n}.in/.out` → 通用 `*.in/*.out`

输出题仅 `.out`/`.ans` 文件时自动生成空输入 `_empty.in`

### 7.6 后端连接管理

**Watchdog 协程**（`main.py`，每 2s）：
- `GET http://localhost:8001/health`（评测机的 `/health` 端点）
- 成功 → 更新 `last_heartbeat`、`status='idle'`
- 失败一次即断开 → `is_connected=False`、所有 `running` 提交 → `system_error`、清空 `_pending_jobs`

**Dispatch 协程**（`judge_service.py`，每 2s）：
- 检查 `is_connected` 且 heartbeat ≤ 30s
- 取一条 `status='pending'` 提交（排除有延迟文件操作的题目，`is_test_run DESC, submitted_at ASC`）
- 读测试数据文件 → HTTP POST 发给评测机 → 注册到 `_pending_jobs`
- 轮询 `_pending_jobs` 中已完成的任务（GET `/job/{job_id}`）→ `_write_result` 回写 DB
- 检查并执行延迟文件操作（该题 running=0 时）

### 7.7 JudgeState 表

单行记录（`id=1`），由 ORM `Base.metadata.create_all()` 自动创建：

| 字段 | 类型 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | INT PK | 1 | 固定为 1 |
| `is_connected` | BIT | 0 | 管理员是否已连接 |
| `last_heartbeat` | DATETIME NULL | - | 最后心跳时间 |
| `status` | VARCHAR(20) | 'stopped' | 评测机状态（stopped/idle/running/waiting，后两者预留未用） |

## 九、业务规则详解

### 9.1 题目状态流转

```
用户上传 → pending_new
           ├─ 管理员审核通过 → approved（分配 problem_number）
           ├─ 管理员拒绝 → rejected
           ├─ 管理员冻结 → frozen
           └─ 管理员归档 → archived

approved 修改测试数据 → pending_modify
           ├─ 管理员审核通过 → approved（保留 problem_number）
           └─ 管理员拒绝 → 回退为 approved（保留原数据不变）

rejected 用户重新提交 → pending_new

pending_modify 管理员仅拒绝修改 → approved（保留原 problem_number）
```

### 9.2 题目可提交条件

```python
can_submit = problem.problem_status in ("approved", "pending_modify") \
    or (is_admin and problem.problem_status == "pending_new")
```

- `approved`：任何人可提交（使用正式测试数据）
- `pending_modify`：任何人可提交（使用正式测试数据，因为 pending 数据待审核）
- `pending_new`：仅管理员可提交（`use_pending=true`，使用待审核数据测试）

### 9.3 评测结果判定流程

1. 编译错误 → `compile_error`
2. 运行超时 → `time_limit_exceeded`（`proc.wait(timeout=time_limit_s)` 抛异常）
3. 内存超限 → `memory_limit_exceeded`（监控线程 kill）
4. 非零退出码 → `runtime_error`
5. 答案文件不存在 → `system_error`
6. 输出比对：
   - normalize（去行尾空格 + 去末尾空行）→ 完全一致 → `accepted`
   - token 一致（split 后相同）→ `presentation_error`
   - 否则 → `wrong_answer`

### 9.4 提交记录可见性

```python
# 列表（list_submissions）
普通用户：is_test_run = 0
管理员：可筛选 is_test_run

# 详情（get_submission_detail）
可查看条件：自己的提交 OR 管理员 OR 题目上传者 OR 已通过该题
```

### 9.5 账号封禁效果

1. `status = 'banned'`
2. `token_version += 1`（所有已签发 Token 立即失效）
3. 当前活跃的轮询请求在下一次 API 调用时收到 401 而停止
4. 被封禁用户无法登录（`user.status == "banned"` → 401）
5. 被封禁用户不显示在排行榜中（`WHERE status = 'active'`）

### 9.6 题目编号分配

```python
max_num = db.query(func.max(Problem.problem_number)).scalar()
problem.problem_number = (max_num or 1000) + 1
```

- 起始编号：1001（若无已分配编号）
- 仅审核通过时分配
- 修改样例后重新审核保留原编号（`problem_number` 已非 NULL 则跳过分配）
- 拒绝时不分配

### 9.7 用户统计更新时机

- `Users.total_submissions`：正式提交（`is_test_run=0`）评测完成后 +1（仅限用户尚未 AC 该题时；已 AC 该题后的重复提交不计数）
- `Users.solved_problems`：首次 AC 某题时 +1（插入 UserAcceptedProblems 时）
- `Problems.accepted_user_count`：首次 AC 时 +1
- `Problems.submissions_before_accepted`：正式提交评测完成后 +1（仅限用户尚未 AC 该题时；与 `total_submissions` 同步更新）
- `Tags.problems`：添加/删除标签关联时重新计数

### 9.8 延迟文件操作（Deferred File Ops）

**问题**：管理员审核通过题目时需移动测试数据文件（pending→正式目录），但若该题仍有 `running` 状态的提交，评测机正在读取这些文件，移动操作会导致评测失败。

**解决**：
1. 审核时将操作注册到全局字典 `_deferred_ops[problem_id] = (action, args)`
2. 评测调度器每轮 `_poll()` 开始时调用 `_try_execute_deferred()`
3. 检查该题 `running` 提交数是否为 0
4. 为 0 则执行：`approve_test_data`（移动）或 `reject_test_data`（删除）
5. 调度器跳过有延迟操作待执行题目的 pending 提交

## 十一、安全性设计

### 11.1 密码安全

- bcrypt 哈希存储（passlib `CryptContext(schemes=["bcrypt"])`）
- 密码长度要求：注册 6-32 位，修改密码同样 6-32 位

### 11.2 Token 安全

- JWT 签名算法：HS256
- Token Payload 最小化原则：仅存 `user_id`、`role`、`token_version`、`type`、`exp`
- 不存用户名等可变信息
- `token_version` 版本号机制实现单设备登录和封禁即时生效
- 所有敏感修改操作（角色、状态）使用 Query 参数传递而非 JSON Body

### 11.3 文件上传安全

- 所有上传文件重命名（UUID），不使用用户提供的文件名
- 文件类型白名单校验（扩展名 + MIME type）
- 文件大小限制
- 测试数据文件名长度限制（≤3 字符，过滤过长文件名）

### 11.4 XSS 防护

- Markdown 渲染使用 DOMPurify 清洗
- KaTeX 生成的 MathML 标签通过自定义 hook 添加到 DOMPurify 白名单
- 后端所有返回的 HTML 内容不直接拼接，由前端渲染

### 11.5 输入校验

- 后端 Pydantic Schema 校验（用户名正则、密码长度、邮箱格式）
- 前端双重校验（提交前校验 + 后端返回错误提示）
- SQL 注入防护：使用 SQLAlchemy ORM（参数化查询），不拼接原始 SQL

### 11.6 代码执行安全（评测机）

- 用户代码在独立子进程中运行
- 时间限制（`proc.wait(timeout)`）
- 内存限制（监控线程，超限 `proc.kill()`）
- 所有文件操作在临时目录中完成，评测后删除
- `subprocess.Popen` 不使用 `shell=True`
- Windows 错误报告弹窗已禁用（`SetErrorMode(0x8003)`）

## 十四、总结与评价

### 14.1 项目总结

本项目是一个功能完整的**算法竞赛在线评测系统（QuickOJ）**，采用前后端分离 + 独立评测机的三层架构。系统涵盖用户管理、题库管理、代码提交与自动评测、题目审核流程、消息通知、排行榜六大核心模块，共计 **37 个 API 端点**、**15 条前端路由**、**8 张数据库表**、**2 个视图**、**24 条索引**。

**技术架构**：SQL Server（数据层）→ FastAPI + SQLAlchemy（业务层）→ Vue 3 + Vite（展示层），评测机作为独立 FastAPI HTTP 服务运行，与后端通过 multipart/form-data 传输代码和测试数据。

**认证体系**：基于 JWT 的无状态认证，搭配 `token_version` 版本号机制实现单设备登录和即时封禁，无需 Redis 黑名单中间件。

**评测链路**：后端后台协程每 2 秒轮询 DB 中的 pending 提交 → 打包测试数据文件 → HTTP POST 发送给评测机 → 评测机在临时目录中编译运行 → 轮询获取结果 → 回写 DB。评测机零数据库依赖、零持久化、进程池隔离。

### 14.2 优点

1. **架构清晰、职责分明**：三层架构中每层职责单一。评测机独立为纯计算 HTTP 服务，不引入后端代码、不访问数据库、不持久化文件，具备良好的可测试性和跨机器部署能力。

2. **数据库设计规范**：全部 8 张表满足 BCNF 范式，CHECK 约束、UNIQUE 约束、外键约束（除有意不设者外）完整覆盖；通过过滤索引和反范式冗余进行有节制的性能优化；数据库视图封装常用查询语义。

3. **安全性多层级防护**：密码 bcrypt 哈希 → JWT + token_version 认证 → ROLE_LEVEL 权限校验 → 文件上传白名单 + UUID 重命名 → DOMPurify XSS 清洗 → 评测机子进程隔离 + 时间/内存双重限制。纵深防御覆盖从 Web 层到执行层的全链路。

4. **边界条件处理细致**：
   - 审核延迟文件操作（等 running 清零再移动/删除）防止评测中断
   - 评测机连接/断开时的状态清理（running → system_error、清空 _pending_jobs）
   - Submissions 表不设外键以保留删除用户/题目后的历史记录
   - 调试提交不创建 Submission 记录、不消耗提交次数

5. **前端交互体验**：响应式布局（移动端汉堡菜单）、实时状态轮询（提交详情、评测机管理、提交列表）、页面可见性检测（隐藏时暂停轮询）、Token 自动刷新队列化（避免并发刷新）、头像本地预览 + 上传中动画。

6. **可部署性**：前端通过 Vite 环境变量切换 API 地址，一份构建产物适配开发/生产环境。后端和评测机通过环境变量配置数据库连接、评测机地址、数据目录，支持本地开发和云部署。

### 14.3 不足与改进建议

1. **评测机并发瓶颈**：当前评测调度器单协程串行取 pending 提交（每次一行），评测机进程池上限为 CPU 核数 - 1，在高并发提交场景下可能出现评测排队积压。
   - **建议**：引入批量取提交 + 评测机内部队列预取机制；支持多评测机实例，通过 JUDGE_SERVER_URL 列表轮询分发。

2. **前端状态管理可增强**：当前仅 Auth 和 Notification 两个 Pinia Store，题目列表的筛选条件、提交详情等状态均为组件局部状态，跨页面共享不便。
   - **建议**：增加 Problem Store（缓存列表筛选条件）和 Submission Store（缓存当前查看的提交）。

3. **缺少接口限流**：当前 API 未实施速率限制，注册、登录、提交等接口可能被恶意高频调用。
   - **建议**：引入 `slowapi` 或基于中间件的 IP/用户级别限流，至少对注册（防批量注册）、提交（防恶意刷评测）实施保护。

4. **评测机错误处理粒度**：运行错误（RE）仅记录退出码和 stderr 文本，未区分段错误（SIGSEGV）、浮点异常（SIGFPE）、非零返回等不同原因。
   - **建议**：在评测机中根据 `proc.returncode`（如 -11 = SIGSEGV）映射为更细粒度的错误原因，并返回给前端展示。

5. **缺少监控与日志**：后端和评测机仅使用 `print()` 输出，无结构化日志，不便于排查问题。
   - **建议**：引入 Python `logging` 模块，按请求 ID 串联日志，将评测调度器状态定期写入 DB 或 Prometheus 指标。

6. **题目搜索能力有限**：当前题库列表仅支持题名模糊搜索和标签筛选，不支持全文搜索题面内容。
   - **建议**：可利用 SQL Server 的全文索引（Full-Text Search）对 `statement` 列建立索引，支持按题面关键词搜索。

7. **提交代码未持久化到对象存储**：源代码存储在数据库 `NVARCHAR(MAX)` 列中，长期高增长场景下表体积变大影响查询性能。
   - **建议**：超过一定阈值的代码文本存储到本地文件或 MinIO，Submissions 表中仅保留路径引用。

8. **缺少自动化测试**：项目中未发现单元测试或集成测试代码。
   - **建议**：为后端 API 添加 pytest + httpx 测试用例，为核心业务逻辑（评测调度、审核流程、权限校验）添加覆盖率保护。

### 14.4 总体评价

本项目是一个**设计完整、实现规范、可直接投产**的算法竞赛在线评测平台。在数据库设计（BCNF 规范化 + 有节制的反范式优化）、系统架构（评测机完全解耦）、安全防护（纵深防御）、前端体验（轮询 + 可见性检测 + Token 自动刷新）等方面均达到了生产级质量。代码组织清晰，业务逻辑与基础设施分离合理，具备良好的可维护性和可扩展性。

上述改进建议属于**工程化增强**方向（监控、限流、测试、全文搜索），不影响当前系统的功能完整性和正确性。以现有设计为基础，本系统可支撑日均数万次提交的中等规模平台稳定运行。若需扩展至竞赛级别的高并发场景，可从评测机集群化、消息队列替代轮询、读写分离等方向进一步演进。
